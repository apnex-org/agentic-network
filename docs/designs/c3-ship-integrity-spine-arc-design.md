## Problem

The autonomous stint was bitten three times by the same defect SHAPE — a step on the merge -> deploy -> substrate-read path completed silently-wrong, with no positive signal of success and no loud signal of failure. Instance 1: a deploy stalled because watchtower's auto-pull raced refresh-docker-token.service (bug-107; 30-min systemd timer vs AR-token TTL) and prod silently froze on a stale image, recovered TWICE only by the sanctioned `systemctl start refresh-docker-token.service` nudge (DR-008); the org's own roll-confirm signal (toolSurfaceRevision, hub-networking.ts:868-876) is an ETag over the MCP tool surface that does NOT change for internal-only deploys, so it false-reads 'not rolled' (DR-011/#346). Instance 2: a partition-relocated FILTERABLE field whose renameMap entry was omitted produced a SILENT filter-miss (querying the wrong JSONB path) while decodeEnvelopeToFlat's blind flat-spread (shape-helpers.ts:90-108) still returned plausible rows, so the read LOOKED correct (bug-138, bug-170-layer2); cal-84's degrade-then-poll-recover catch even swallowed the read-crash. Instance 3: squash-merge-then-continue-branch staleness nearly DELETED merged docs (DR-011), rescued only by a hand cherry-pick onto a fresh branch off main; the require-up-to-date rule also forces a serial-rebase treadmill + a keyring self-approve dance. HONEST SCOPE (cal-81 guard): these are THREE observed INSTANCES of a broader class. This arc makes those three fail-loud + adds positive-completion contracts + stands up two general governors (a renameMap-governor over new-kind filterable fields; a content-deletion governor). It does NOT prove the whole silent-failure-on-the-path-to-prod CLASS is universally closed — above-membrane consumer mis-reads, a shallow /health that passes on a runtime-broken image, and non-delete content corruptions remain uncovered and are named per-rung.

## Approach

Govern C3 as a PORTFOLIO of parallel banks with named owners, bound by ONE cross-cutting methodology charter — the No-Silent-Completion contract — NOT a serial L-spine (the 'spine' is a narrative wrapper, not an engineering dependency graph). The charter imposes, for every governed step, a two-part obligation: (1) a POSITIVE completion signal asserted by a writer that SURVIVES the step, and (2) a FAIL-LOUD signal that alarms when the positive signal is absent or mismatched within an SLA. Three banks map to the three stint instances + a capstone codifies the charter: deploy bank (R1->R2->R3, serial WITHIN the bank), substrate bank (R4, own owner), integration bank (R5, near-zero coupling), methodology capstone (R6). Three structural invariants drive every mechanism. INVARIANT A — OBSERVER PROBLEM: the Hub IS the artifact being rolled, so /health and any in-Hub DeployState writer live on the thing that restarts. Therefore /health only EXPOSES identity; an EXTERNAL writer that survives the restart (deploy-hub.yml CI-side + R2's VM-side/external prober) ASSERTS roll-confirm, health-probe, and DeployState terminal status; a Hub-deploy stall alarms from OUTSIDE (R2), and the prober's OWN liveness is a charter line-item (who-watches-the-watcher: dead-man's-switch + a second independent alarm path, because a GitHub scheduled-cron prober silently auto-disables after 60d and is delayed under load). INVARIANT B — GOVERNANCE-GATED AUTONOMY: four surfaces reach for prod-mutation/approval authority the org keeps human-gated (R2 event-triggered roll, R2 STANDING freshness-guard auto-nudge, R3 unsupervised auto-rollback, R5 auto cross-approval). DR-008 blessed the refresher nudge 'in this instance', NOT as standing authority; PR approvals route peer-to-peer. Each ships ONLY its mechanism in alarm + asserted/one-click form NOW; all four are BATCHED into ONE Director-ratification governance proposal BEFORE any auto path is built, so adoption is monotonic-safe (every rung is standalone-valuable in its human-gated form even if no autonomy is ever ratified). INVARIANT C — HONEST SCOPE: each rung names what its gate does and does NOT cover; green gates are scoped, never claimed as class-elimination. Two cross-bank edges are real and deliberate: (1) R3's new DeployState kind is sequenced UNDER R4's code-derived renameMap completeness gate as the governor's FIRST DOGFOOD, closing the R4->R3 self-coupling the arc's own dep-graph missed; (2) R6 soft-depends on R1/R3/R4. The deploy bank's terminal rung (R3) is therefore gated cross-bank on R4 — stated, not oversold as 'fully parallel'. D-1 RESOLUTION: the D-1 sovereign control plane (root cli/api) is verified UNBUILT (no cli/ or api/ dir exists), so 'ois-deploy is a facet of the sovereign plane, not a one-off CLI' is a hidden cross-arc dependency. Resolve via a FACET-SHIM contract: get_deploy_status ships now as an MCP verb (that surface exists); ois-deploy is a thin shim born inside the plane's resource(DeployState)/verb(status,watch,verify) model, designed to slot into D-1 when it lands — a SOFT-gate, with the soft-vs-hard choice surfaced to the Director. Verifier (steve) engages at the structural-elimination claims per cal-83 deep-adversarial-audit + cal-79/82 faithful-prod-shape harness + cal-80 value-not-count + cal-81 no-subset-extrapolation; advisory, never gating (Director stays Accountable at every gate). Calibration filings (cal-84 closure, partition-renameMap promotion) route architect-Director per CLAUDE.md ledger discipline, NEVER LLM-autonomous.

## Adversarial verdict (the verify pass)

**Holds:** False

**Weaknesses:**
- GOAL OVER-CLAIM (primary, triggers the org's own cal-81 'premature structural-closure from partial verification'): the headline 'structurally ELIMINATE the silent-failure-on-the-path-to-prod CLASS' is not conclusively met. What the rungs deliver is fail-loud + positive-completion contracts for the THREE observed instances (deploy-stall, renameMap filter-miss, phantom-delete) plus two general governors (R4 gates every new kind's FILTERABLE fields only; R5 gates content-deletion only). The class is broader: arbitrary above-membrane consumer mis-reads, a shallow /health that passes on a runtime-broken image, and non-delete content corruptions are all uncovered. Only Draft 3 states this honestly per-rung; Drafts 1 and 2 carry the class-elimination framing forward, which is the exact pathology cal-81 names.
- C8 DEPENDS ON AN UNBUILT PLANE: the constraint 'all new surfaces expose THROUGH the D-1 sovereign control plane (root cli/api)' is load-bearing, but verified there is NO root cli/ or api/ directory in the repo — D-1 is a separate directive still in design. So R3's 'ois-deploy is a facet of the sovereign plane, not a one-off CLI' is a hidden cross-arc dependency: either R3 waits on D-1 (contradicting the 'OPENER, buildable now, no feasibility gate' claim) or it ships a standalone CLI (violating C8). None of the three drafts surface or resolve this tension; all three simply assert facet-hood.
- R1 gitSha-bake mechanism unsolved in Drafts 1 and 3: build runs via 'gcloud builds submit --tag' (build-hub.sh:153 = Cloud Build), which has no local --build-arg, so 'add running git-SHA to /health' is not free. Only Draft 2 grapples with it (generate hub/build-info.json into the Cloud Build context, COPY in Dockerfile, /health reads it; digest stays external because the registry computes it post-push). Drafts 1/3 hand-wave the bake step, the load-bearing part of the same-day S rung.
- Draft 1 factual error: states 'entity-kinds.json LOCKED 20->21'. Verified the file is 22 kinds at schemaVersion 2.1, so DeployState is 22->23 (Drafts 2 and 3 are correct). Drafts also disagree on the count, signalling none cross-checked the locked inventory.
- cal-84 closure-filing discipline missed by Drafts 2 and 3: verified cal-84 is OPEN, architect-authored ('OPEN until the 0-bare anomaly-monitor lands', closure_pr #324). CLAUDE.md mandates ledger filings/closures are architect-Director, never LLM-autonomous. All three correctly specify the cal-84 MECHANISM (fail-loud re-throw OR 0-bare/anomaly monitor) matching the ledger, but only Draft 1 fences the ledger CLOSURE as architect-Director-routed; Drafts 2/3 leave a mission free to self-close it.
- R1 roll-confirm SLA vs watchtower cadence: a blocking poll job inside deploy-hub.yml (Draft 2) ties up CI minutes and must out-wait watchtower's own poll interval + image-pull. It conflates a legitimately slow roll with a stalled roll (bug-107), so it will false-alarm on latency or time out before a slow-but-healthy roll completes. The SLA is asserted ('e.g. 5 min') without anchoring to the watchtower poll cadence; an external survivor-prober is a better home for roll-confirm than a blocking CI job.
- External backstop has its own silent-stop mode: Draft 2's external prober is a GitHub scheduled-cron workflow (hub-liveness.yml, ~10 min). GitHub scheduled workflows auto-disable after 60 days of repo inactivity and are delayed/dropped under load — silently. Using it as THE deploy-stall backstop re-opens the who-watches-the-watcher problem the arc explicitly worries about. Draft 1 names the residual but doesn't make the prober's own liveness concrete; Drafts 2/3 don't flag GitHub-cron-specific failure modes.
- R5 branch-hygiene gate false-positive surface: blocking 'any PR deleting/reverting main content outside its intended diff' will fire on legitimate intentional deletions and file-moves. Even Draft 2's precise definition ('net-removes a file present on main the PR's own commits never modified') false-positives on a pure-delete PR. No intended-delete override (label/trailer) is specified, so the gate risks becoming a friction source the arc is supposed to reduce.
- R4 harness-fidelity under-specified in Draft 1: the round-trip oracle must run on real Postgres JSONB filtering (cal-79/82 faithful-harness is non-negotiable) and assert VALUE round-trip, not count-parity (cal-80: filter-count passes while rows are still raw envelope). Draft 2 specifies testcontainers + value round-trip; Draft 1's vaguer 'behavioral backstop' risks the very false-green it is meant to prevent.
- 'Parallel banks' is partly oversold: R3 hard-blocks on R4's gate and R6 on R1/R3/R4, so the deploy bank (R1->R2->R3) cannot complete independently of the substrate bank. All three self-disclose this as a risk, but the headline 'four parallel banks' undersells that the deploy bank's terminal rung is gated cross-bank.
