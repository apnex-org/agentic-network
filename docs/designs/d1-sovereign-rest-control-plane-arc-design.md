## Problem

Today the Hub's ONLY external surface is the MCP relay — the dataplane wire protocol. A CLI can reach the Hub only by dogfooding /mcp (get-agents.sh does the initialize-handshake + SSE-strip + jq-unwrap dance). There is no REST API and no root cli//api/ folders. The Director clarified at the stint-2 gate (2026-06-21) that the long-term D-1 control plane MUST be a first-class REST API AT THE HUB (Hub = kube-apiserver), a robust modular CLI that calls that REST API (oisctl = kubectl), DECOUPLED from MCP, and AGENT-DRIVABLE as a first-class use (agents = controllers). The naive framing — 'both bindings call router.handle(), so REST inherits RBAC/audit for free' — is FALSE as written and is the load-bearing trap of this whole arc: (1) router.ts:150 enforces RBAC only when the resolved role != 'unknown', so a synthetic-session REST request resolves 'unknown' and FAILS OPEN — strictly MORE privileged than the same call over MCP, violating the hard 'no-more-privileged-than-MCP' + tighten-only constraints; (2) RBAC reads engineerRegistry.getRole(ctx.sessionId), a session-keyed side-table, NOT ctx.role — so a token→role binding does not gate RBAC unless the seam is changed; (3) resolveCreatedBy reads getAgentForSession(ctx.sessionId) for agentId, so a sessionless REST create is attributed to anonymous-<role>, an audit/provenance gap (tele-4/tele-12); and getting a real agentId triggers the ADR-021 auto-claim displacement (router.ts:177) that would corrupt a live MCP session. Full RBAC + full provenance + no session-displacement cannot all hold under the current code without one bounded seam change. The design must EARN the control-plane label rung-by-rung, make 'one authority, two bindings' structural (not a duplicated-logic aspiration), and never break the live MCP dataplane the in-harness adapters depend on.

## Approach

Charter-first, risk-sequenced, record-honest. The de-risking spine: the Hub ALREADY separates AUTHORITY from BINDING — PolicyRouter.handle() (router.ts:134) is the apiserver core (verb registry + RBAC + ADR-021 auto-claim + audit + cascade dispatch), and bindRouterToMcp (mcp-binding.ts, 76 lines) is already a thin policy-free projection that iterates router.getAllToolNames()/getToolRegistration() and forwards to router.handle(). REST becomes a sibling bindRouterToRest mounted on the SAME Express app (already serving /mcp + /health), terminating in the SAME router.handle(). The dual-binding INVARIANT (Draft 1's strongest contribution) is mechanized as a charter: EVERY binding terminates in router.handle(); NO binding re-implements RBAC, audit, dispatch, or a hand-listed route table. But — and this is the keystone the drafts missed — 'inherits RBAC for free' is only true AFTER a bounded identity-seam consolidation: thread a resolved {role, agentId} onto IPolicyContext and have router.handle's RBAC + resolveCreatedBy consume ctx-first (engineerRegistry session-lookup as fallback). This (a) closes the fail-open hole by making REST always resolve a CONCRETE role (never 'unknown'), (b) makes the token→role binding actually gate RBAC, (c) gives real agentId provenance, and (d) incidentally fixes the documented ctx.role-vs-getRole drift (bug-168/169). It is additive at the binding layer but ONE bounded change at the identity seam — framed honestly, not as 'zero authority code'. The exploitable seam: setSessionRole (agent-repo-substrate.ts:284) writes a sessionRoles map INDEPENDENT of currentSessionId, so REST can seed correct RBAC WITHOUT triggering ADR-021 displacement — the mechanism that makes 'RBAC parity without session split-brain' achievable. Sequencing puts SAFETY before LEVERAGE: a not-banked charter+identity-feasibility spike (R0) resolves the deepest risk and the REST↔MCP Director question before any code; a READ-ONLY REST binding (R1) proves the derived-contract + binding machinery at zero mutation hazard, honestly named a 'REST read surface' not a control plane; the identity-seam rung (R2) makes REST safely write-capable with an IDENTITY-DRIVEN parity gate (wrong-role→DENY, unknown→DENY — the regression that pins the fail-open hole shut); the write/actuate rung (R3) EARNS the control-plane label and discharges the agentic dogfood; the kubectl-analog oisctl (R4) is a thin REST client decoupled from /mcp; the agentic-drive loop + WATCH (R5) makes agent-as-controller first-class; and a not-banked convergence register (R6) names the perfect-long-term destination as DEFERRED rather than over-building it. The contract is DERIVED at runtime by generalizing the computeToolSurfaceRevision walk (tool-surface-revision.ts:62) — resources = kinds that have registered policy verbs (which auto-EXCLUDES the 5 infra kinds and auto-CORRECTS the verified 20-vs-22-vs-23 drift), folded with the SchemaDef inventory for fields/watchable/renameMap — never hand-listed. The projection CONVENTION (kind→resource, CRUD→method, action→subresource, WATCH→SSE-over-LISTEN/NOTIFY) is defined HERE at architecture altitude; only exact resource/verb STRINGS defer to idea-121. C2/C3/C4 verbs SLOT IN as registered tools projected through this plane but are chartered-OUT and built in their own arcs.

## Missions (banked / staged rungs)

### R0 — M-Control-Plane-Charter-And-Identity-Seam-Spike
_size S · not-banked_

NOT-BANKED gate (architect-owned). Front-loads the deepest risk — the auth/identity decoupling — and resolves the REST↔MCP relationship as the mandated Director question BEFORE any code. Deliverables: (1) THE DUAL-BINDING CHARTER (docs/specs/ois-control-plane-charter.md): names PolicyRouter.handle() as the single authority, MCP+REST as bindings, and the load-bearing invariant — every binding terminates in router.handle(); NO binding re-implements RBAC/audit/dispatch or hand-lists routes; a new kind/relocated-field/verb appears on BOTH bindings from the one registry+SchemaDef walk; forking authority (e.g. a raw-substrate REST endpoint) is a charter violation. (2) THE NAMING GATE + over-claim guard (4 properties: REST-ACTUATION verbs exist; SINGLE-AUTHORITY via shared router.handle; RBAC+AUDIT byte-for-byte PARITY, REST no-more-privileged-than-MCP; AGENTIC-DRIVE dogfooded) — a read-only REST surface is a STATUS PAGE and may NOT be called a 'control plane' until the write/actuate rung discharges all four. (3) THE IDENTITY-SEAM VERDICT confronting three verified code facts: unknown role FAILS OPEN (router.ts:150); RBAC reads getRole(sessionId) not ctx.role (router.ts:149); resolveCreatedBy reads getAgentForSession(sessionId) for agentId, falling back to anonymous-<role> (caller-identity.ts:53,63). Recommend the bounded seam consolidation (thread {role,agentId} onto IPolicyContext; RBAC + resolveCreatedBy consume ctx-first with registry fallback — also fixes bug-168/169) + token-bound REST identity that always resolves a concrete role (fail-CLOSED) and uses setSessionRole WITHOUT claimSession to avoid ADR-021 displacement. (4) THE PROJECTION CONVENTION (docs/specs/ois-api-conventions.md, architecture altitude): kind→resource, CRUD→method/path, action→POST /<id>/<action> subresource, list/filter→GET collection via renameMap field-paths, WATCH→SSE over LISTEN/NOTIFY; resources derived from verb-registry-backed kinds, never hand-listed; PolicyResult→HTTP unwrap contract (content[0].text|>JSON.parse→body; isError→4xx/5xx; non-JSON prose→{message} flagged for idea-121). (5) THE D-1 ROOT-LAYOUT DECISION (api/ = contract+conventions+OpenAPI+conformance; cli/ = oisctl; runtime binding in hub/src/rest/). (6) THE IDENTITY-DRIVEN CONFORMANCE-GATE SPEC (correct-role→allow+audit; wrong-role→DENY; unknown/no-identity→DENY). No production code ships.

### R1 — M-REST-Read-Binding-And-Derived-Contract
_size M · banked_

BANKED, the apiserver skeleton, deliberately READ-ONLY for safety. Ships bindRouterToRest (hub/src/rest/, twin of mcp-binding.ts) on the existing Express app — no new deployment surface on the VM/watchtower container. Projects ONLY read verbs (get/list/filter) as GET /apis/core.ois/v1/<resource>{/id}, plus GET /apis/core.ois/v1 discovery (api-resources + explain) generated from the DERIVED contract: deriveContract(router, schemaDefs) generalizes the computeToolSurfaceRevision walk (tool-surface-revision.ts:62) to emit resources = kinds that have registered policy verbs (auto-excluding the 5 infra kinds Counter/MigrationCursor/RepoEventBridgeCursor/RepoEventBridgeDedupe/SchemaDef, and auto-correcting the verified 20/22/23 drift), folding SchemaDef fields/watchable/renameMap. ETag reuses computeToolSurfaceRevision. CRITICAL fail-closed: even read requests must resolve a CONCRETE role (transport bearer reused from bearer-auth.ts; no synthetic 'unknown' session reaches handle()) — reads still flow through router.handle so read-audit + RBAC are identical to MCP. Creates root api/ (conventions doc + CI-diffed generated OpenAPI snapshot). Honestly a 'REST read surface', NOT a control plane (naming gate). Reversible: additive routes behind REST_API_ENABLED; /mcp untouched; code-only watchtower redeploy. Standalone value: org-state readable over plain HTTP/curl (the 'agents read org-state' leg) with a self-correcting contract, zero mutation hazard.

### R2 — M-REST-Identity-RBAC-Audit-Parity
_size L · banked_

BANKED keystone — the genuinely hard rung that makes REST safely WRITE-capable. Implements the R0 identity-seam verdict: thread a resolved {role, agentId} onto IPolicyContext; router.handle RBAC (router.ts:149) and resolveCreatedBy consume ctx-first with the engineerRegistry session-lookup as fallback (one bounded consolidation of the authority seam, not a fork; incidentally fixes bug-168/169). Agent-drivable credentials = token-bound role (extend the mission-86 TokenStore/bearer_tokens with a nullable token→agentId+role binding — additive migration; legacy HUB_API_TOKEN grandfathered; flagged as a cross-mission dependency requiring coordination). A presented bearer resolves to a CONCRETE {role, agentId} so RBAC fires identically to MCP and is tighten-only (a token can never grant more than its role); REST seeds setSessionRole on a synthetic session WITHOUT claimSession (agent-repo-substrate.ts:284 seam) so no ADR-021 displacement of any live MCP session. Delivers the IDENTITY-DRIVEN PARITY CONFORMANCE GATE (the regression that pins the fail-open hole shut): for representative verbs — correct-role→ALLOW + same audit row as MCP; wrong-role→DENY; unknown/no-identity→DENY; every tool projected (none silently dropped); every REST route backed by a registry tool. Standalone value: an authenticated, RBAC-scoped, fully agent-attributable REST surface — still ZERO verbs beyond what MCP exposes (one authority preserved); unlocks safe mutation.

### R3 — M-REST-Write-Actuate-Binding
_size L · banked_

BANKED — EARNS the control-plane label. With identity+RBAC+audit parity proven (R2), projects the mutating + ACTION surface via router.handle: POST /apis/core.ois/v1/<resource> (create), PUT/PATCH (update), DELETE, and ACTION verbs (claim/lease/ack/actuate/...) as POST /apis/core.ois/v1/<resource>/<id>/<action> subresources (the kubectl subresource analog per the R0 convention). Because every route is a thin projection onto router.handle, the cascade/FSM/CAS transforms ride for free; PolicyResult unwraps to clean JSON bodies per the R0 contract. THE AGENTIC-DRIVE DOGFOOD discharges the charter naming gate: lily executes a real lifecycle actuation through REST end-to-end, proving ACTUATION + SINGLE-AUTHORITY + PARITY hold. Only at this rung does the surface legitimately become a 'control plane'. Idiomatic resource-noun REST (not a POST /:verb RPC tunnel) per the convention defined in R0 — exact resource/verb STRINGS defer to idea-121, the convention does not. Reversible: additive write routes behind REST_API_ENABLED; MCP write path untouched. Standalone value: a sovereign, audited, RBAC-scoped REST control plane agents and operators actuate the org through.

### R4 — M-oisctl-Modular-CLI
_size L · banked_

BANKED — the kubectl-analog 'robust modular flexible' CLI, a THIN client of the REST API, DECOUPLED from /mcp (the explicit override; the prior extend-mod.core+dogfood-/mcp lean is RETIRED). Lives at root cli/. oisctl <resource> <verb>; discovery (api-resources/explain) generated from the R1 derived contract (not hardcoded); output formats table/json/yaml/jsonpath (fold mod.core buildTable() + tpl/*.jq into cli/oisctl/lib/render); config + contexts (~/.config/oisctl/config — identity token + multi-Hub/env, the kubeconfig analog consuming R2 credentials); a plugin/extensibility model (scoped, not over-built); scripting + agentic ergonomics. FOLD the scattered scripts into oisctl subcommands and demote originals to one-way shims at cutover (no two live writers): get-*.sh→oisctl get, calibrations.py READ-verbs→oisctl calibration list/show (calibration WRITE-verbs stay architect/Director-authored, NOT folded), hub-snapshot.sh→oisctl snapshot. get-entities.sh direct-psql STAYS the named un-sovereign BREAK-GLASS (psql bypasses RBAC+audit — must never present as a sovereign verb). Recommend a thin REST-native TS client reusing Hub zod/types (single language with the Hub) over a from-scratch compiled binary; Go-binary + plugin marketplace chartered-deferred. Depends on R1-R3. Standalone value: one sovereign CLI over the audited REST plane replaces the drift-prone script scatter.

### R5 — M-Agentic-Drive-Loop-And-Watch
_size M · banked_

BANKED — makes 'an agent drives the org via REST as fluently as via MCP today' a proven, first-class use-case. Delivers the agentic-drive CONTRACT (the controller pattern: read org-state via R1, authenticate as an agent identity via R2, actuate lifecycle via R3 verbs surfaced on the plane) + WATCH. WATCH = GET /apis/core.ois/v1/<resource>?watch=true as SSE (reusing the proven GET /mcp SSE plumbing over the mission-83 substrate-watch LISTEN/NOTIFY primitive in postgres-substrate.ts — NEVER an FS-walk/DB poll loop; honors the bug-93 poll-pressure structural elimination), long-poll as a NOTIFY-backed fallback, gated on SchemaDef watchable:true; true list/filter result sets translate flat filter keys to envelope paths via each kind's renameMap (the single field-path authority; tele-12). The PRIMARY agentic-drive use-case is the C1 work-queue (work-item kind + claim/lease/ack) — but that kind+verbs are C1's keystone, BUILT in C1 and merely CONSUMED here as registered tools the plane projects (a new kind appears on BOTH bindings by construction; this rung does NOT build the work-queue — no scope-creep into C1). PROOF discipline: demonstrate the read→claim→actuate→evidence drive loop with an OUT-OF-HARNESS controller (script/sidecar over REST), validating first-class agentic drive WITHOUT forcing the in-harness Claude Code / OpenCode adapter off MCP; cross-adapter parity for steve/OpenCode is the known-hard leg, bounded to whatever the C1/C2 W0 spike licenses, never silently proxied. Success bar = an agent completes the loop via REST end-to-end as fluently as via MCP. Standalone value: a language-agnostic agent-as-controller capability with reactive event-watch.

## Success criteria

- A REST request is provably NO MORE privileged than the same operation over MCP: the identity-driven conformance gate asserts wrong-role→DENY and unknown/no-identity→DENY over REST (the router.ts:150 fail-open hole cannot be reached because REST always resolves a concrete role).
- Single authority, two bindings holds structurally: a new kind / relocated field / new verb appears on BOTH the REST and MCP bindings from one walk of the registry + SchemaDef inventory, with no second handler set and no hand-listed route table (conformance: every tool projected, every route registry-backed).
- The resource/verb contract is DERIVED at runtime (resources = verb-registry-backed kinds ∩ SchemaDef inventory); a CI test asserts zero hand-listed kind counts; the 20/22/23 drift is auto-corrected and the infra kinds are auto-excluded.
- Audit + provenance parity: a REST-created entity is attributed to the real agentId (e.g. lily), not anonymous-<role>; every REST mutation emits the same audit row as the MCP equivalent.
- REST never triggers ADR-021 session displacement of a live MCP session (verified: REST seeds setSessionRole without claimSession).
- oisctl drives the org as a thin REST client with ZERO calls to /mcp; api-resources/explain render from the derived contract; the scattered scripts are folded as one-way shims; get-entities.sh direct-psql remains the only named break-glass.
- The agentic-drive success bar is met: an agent completes a read→claim→actuate→evidence loop via REST/oisctl end-to-end, dogfooded out-of-harness, as fluently as via MCP today.
- The live MCP dataplane is untouched throughout: every rung is additive behind REST_API_ENABLED, reversible (flip → MCP-only), shipped as a code-only watchtower redeploy with no big-bang cutover.
- WATCH rides the LISTEN/NOTIFY substrate-watch primitive (SSE) and reintroduces zero poll-pressure (no FS-walk/DB poll loop; bug-93 class not regressed).
- The 'control plane' label is applied only after the R3 write/actuate rung discharges all four charter properties; R1/R2 are honestly named a 'REST read surface' / 'authenticated REST surface'.
- C2 lifecycle / C3 deploy-state / C4 grant verbs demonstrably SLOT IN as registered tools projected via the R0 convention without modifying the binding — but are not built in this arc.

## Risks (synthesis)

- AUTHORITY FORK (deepest): bindRouterToRest grows a hand-maintained route table or parallel RBAC/audit, so a new kind/relocated-field/verb appears on one binding but not the other (the bug-137/138 class made cross-binding). MITIGATE: R0 charter invariant (every binding terminates in router.handle; routes generated) + the R2 identity-driven parity conformance gate that fails on any REST route un-backed by a registry tool or a divergent role decision.
- FAIL-OPEN PRIVILEGE ESCALATION (verified, the trap the drafts missed): a synthetic-session REST request resolves getRole()='unknown' → router.ts:150 BYPASSES RBAC → strictly more privileged than MCP. MITIGATE: REST must always resolve a CONCRETE role (token-bound, fail-CLOSED); the conformance gate's unknown/no-identity→DENY assertion is the regression test pinning it shut.
- IDENTITY-SEAM CHANGE IS NOT 'ZERO CODE': RBAC reads getRole(sessionId) not ctx.role, so a token→role binding does not gate RBAC without threading {role,agentId} onto ctx — a bounded change to the authority seam. MITIGATE: scope it as ONE consolidation rung (R2), frame honestly (additive at binding layer, one seam change), and note it incidentally fixes bug-168/169.
- PROVENANCE-vs-DISPLACEMENT THREE-WAY CONFLICT: full RBAC + real-agentId provenance + no ADR-021 displacement cannot all hold without the seam change (real agentId via getAgentForSession triggers auto-claim displacement). MITIGATE: thread agentId via ctx (no getAgentForSession needed) + setSessionRole-without-claimSession seam; resolved explicitly at R0/R2.
- MCP DATAPLANE BREAKAGE during migration (in-harness lily/greg/steve adapters depend on /mcp). MITIGATE: REST purely additive over the stateless-singleton router; no shared mutable state altered; binding removable behind REST_API_ENABLED; in-harness migration deferred to R6/idea-240; never a big-bang.
- CONFORMANCE-GATE TAUTOLOGY: a surface-equality-only test passes trivially if both bindings call router.handle, proving nothing about RBAC. MITIGATE: the gate is IDENTITY-DRIVEN (correct→allow, wrong→deny, unknown→deny), not surface-derivation-only.
- POLICYRESULT IMPEDANCE: handlers return {content:[{text:JSON.stringify(body)}],isError} (types.ts:82); a naive forward leaks a double-encoded MCP envelope; prose handlers degrade. MITIGATE: the R0 unwrap contract (content[0].text|>JSON.parse→body, isError→status, prose→{message} flagged for idea-121).
- UNDER-DELIVERING IDIOMATIC REST: a POST /:verb RPC tunnel satisfies 'decoupled from /mcp' literally but reads as MCP re-skinned, not the Director's kube-apiserver ask. MITIGATE: the projection CONVENTION (idiomatic resource-noun REST) is defined HERE at architecture altitude; only exact strings defer to idea-121.
- WATCH POLL-PRESSURE REGRESSION: a naive long-poll/DB-scan reintroduces the bug-93 sweeper-poll-pressure (74% Hub CPU) eliminated at mission-83 W5. MITIGATE: WATCH rides the LISTEN/NOTIFY substrate-watch primitive (SSE), long-poll only as a NOTIFY-backed fallback, gated on SchemaDef watchable.
- NEW ATTACK SURFACE: a REST surface on the prod watchtower VM is broader + more discoverable than MCP's stateful handshake; a misconfigured route could expose over-permissive access. MITIGATE: reuse the mission-86 fail-closed bearer gate for all /apis routes; admin routes keep the separate HUB_ADMIN_TOKEN guard; RBAC tighten-only; parity test proves no escalation vs MCP.
- CROSS-MISSION TOKENSTORE DEPENDENCY: the R2 token→agentId+role column touches mission-86 substrate (bearer_tokens/token-store.ts). MITIGATE: nullable additive column defaulting to current behavior; legacy HUB_API_TOKEN grandfathered; flagged as a coordination point, not an incidental add.

## Adversarial verdict (the verify pass)

**Holds:** True

**Weaknesses:**
- SHARED, UNFLAGGED PRIVILEGE-ESCALATION HOLE (all three drafts). The drafts' central de-risking claim — 'both bindings call router.handle(), so RBAC is inherited for free / a REST request is no more privileged than the same MCP call' — is FALSE as written. router.ts:149-159 enforces RBAC ONLY when the resolved role != 'unknown': `const callerRole = ctx.stores.engineerRegistry.getRole(ctx.sessionId); if (!tool.roles.has('any') && callerRole !== 'unknown') {...reject...}`. An unknown caller BYPASSES RBAC entirely (documented back-compat for pre-register_role). A naive REST binding that synthesizes a fresh sessionId not in the in-memory sessionRoles map → getRole() returns 'unknown' (agent-repository-substrate.ts:288-290) → EVERY role-scoped tool is callable → the REST request is STRICTLY MORE privileged than over MCP. This directly violates the hard constraint 'A REST request must be no more privileged than the same operation over MCP' and 'RBAC tighten-only.' Draft 3 comes closest (its R0 spike names that getRole is session-keyed and TokenStore is role-less) but even Draft 3 does not surface that unknown does not merely fail-closed — it actively bypasses, i.e. fails OPEN. None of the three names this.
- RBAC READS ROLE FROM THE STORE, NOT FROM ctx (all three, esp. Draft 2). router.handle's RBAC decision (router.ts:149) reads `ctx.stores.engineerRegistry.getRole(ctx.sessionId)` — a session-keyed in-memory side-table — NOT `ctx.role` (ctx.role is used only for logging, router.ts:260/266). Draft 2's keystone mechanism — 'extend TokenStore with a role column and resolve a presented bearer directly to {role, agentId} in rest-context-factory.ts' — does NOT wire into the RBAC check at all, because the router ignores the context's role and consults the store by sessionId. To make token-bound REST identity actually gate RBAC, the binding must EITHER call engineerRegistry.setSessionRole(syntheticSessionId, role) per request (a write to a shared singleton's in-memory map — leak/concurrency surface) OR change router.handle to source the role from ctx. Either way it is NOT a 'pure additive thin binding with zero authority-layer code' — it is a bounded but real change to the authority seam. Draft 1's 'ZERO/near-zero Hub code' R0 and Drafts 2/3's 'inherits RBAC for free' framing all understate this.
- AUDIT-ATTRIBUTION (provenance) PARITY ALSO BREAKS for a sessionless REST request (all three; the 'audit parity / no visibility gap' tele-4 promise). resolveCreatedBy (caller-identity.ts:32-65) stamps createdBy from getRole(ctx.sessionId) for role AND getAgentForSession(ctx.sessionId) for agentId. For a synthetic, agent-unbound REST session, getAgentForSession returns null → agentId falls back to the placeholder `anonymous-<role>` (caller-identity.ts:63). So REST-created entities would be attributed to 'anonymous-architect', not 'lily' — a real audit/provenance precision gap (tele-4 no-silent-failure, tele-12 precision-context). The deeper tension none of the drafts resolves: getting a REAL agentId requires the agent record bound to the session (getAgentForSession), but binding it is exactly what triggers the ADR-021 auto-claim displacement inside router.handle (router.ts:176-213) that Drafts 2/3 explicitly want to avoid for stateless REST. Full RBAC + full provenance + no session-displacement cannot all hold under the current code without a seam change — and no draft names this three-way conflict.
- POLICYRESULT IMPEDANCE unaddressed by Drafts 1 and 3 (Draft 2 catches it). Handlers return the MCP-shaped envelope `{content:[{type:'text', text: JSON.stringify(body)}], isError?}` (types.ts:80-84; router.ts:138 shows the JSON-stringified-into-text pattern). A 'thin binding' that forwards router.handle output to an HTTP client leaks a double-encoded MCP content envelope, not a clean JSON REST body — and isError must be mapped to an HTTP status. Draft 2 is the ONLY draft that surfaces this (its result-unwrap.ts shim + the 'prose handler degrades to {message:text}' caveat). Drafts 1 and 3 never mention the response shape, so their 'thin sibling binding' is under-specified at exactly the REST-fidelity boundary.
- DRAFT 2 RUNG-1 MAY NOT DELIVER 'REST' IN THE DIRECTOR'S SENSE. Draft 2's first banked rung ships a flat-RPC projection `POST /apis/core.ois/v1/:verb -> router.handle(verb, body, ctx)` and DEFERS the idiomatic kube-resource-noun projection (GET/POST/PATCH/DELETE /{resource}) to idea-121. But the Director's explicit framing is 'Hub = kube-apiserver (REST), oisctl = kubectl' and the override is precisely to reject an MCP-RPC-shaped tactical surface. A `POST /:verb` RPC tunnel over HTTP is arguably MCP-RPC re-skinned with a different envelope — it satisfies 'decoupled from /mcp' literally but not 'a first-class REST API' idiomatically. So Draft 2's FIRST standalone rung risks under-delivering the very thing the override demands.
- CONFORMANCE-GATE TAUTOLOGY RISK (all three, weakest in Draft 1). The 'binding-parity conformance test' is partly tautological: if both bindings call the same router.handle, they trivially expose the same surface and resolve the same RBAC 'by construction' — the test passes without proving anything. The ONLY meaningful parity assertion is per-identity: drive a role-scoped tool over REST with a wrong-role credential and assert DENIAL (i.e. exercise the unknown-bypass hole above). Drafts 2/3 gesture at 'same RBAC decision for the same identity'; Draft 1's gate is framed around surface-derivation parity (same tools projected) and would NOT catch the unknown->bypass privilege hole. The gate must inject real, differing identities to have teeth.

**Improvements (folded into the synthesis above):**
- RESOLVE THE IDENTITY SEAM EXPLICITLY AS THE KEYSTONE, AHEAD OF THE READ BINDING. Adopt Draft 3's front-loaded auth-feasibility spike, but extend its mandate to confront the three concrete code facts: (a) unknown role FAILS OPEN at router.ts:150 (must become fail-closed for the REST path or the REST identity must always resolve a concrete role); (b) RBAC reads getRole(sessionId), not ctx.role; (c) resolveCreatedBy reads getAgentForSession(sessionId) for agentId. The cleanest structural fix (and the one to recommend to the Director) is to make BOTH bindings source identity from ctx: thread a resolved {role, agentId} onto IPolicyContext and have router.handle's RBAC + resolveCreatedBy consume it, with the engineerRegistry session lookup as fallback. This is a small, bounded CONSOLIDATION of the authority seam (not a fork) — and it incidentally fixes the known ctx.role-vs-getRole drift already documented in caller-identity.ts (bug-168/bug-169) and router.ts. Frame it honestly: REST is additive at the binding layer but requires one bounded change at the identity seam — not 'zero authority code.'
- EXPLOIT THE setSessionRole / claimSession DECOUPLING. setSessionRole (agent-repository-substrate.ts:284) writes a separate in-memory sessionRoles map and is INDEPENDENT of agent.currentSessionId (what getAgentForSession + the ADR-021 auto-claim key on). This means a REST request CAN get correct RBAC (seed setSessionRole on a synthetic session) WITHOUT triggering session-displacement (never assertIdentity/claimSession it). State this seam explicitly as the mechanism that makes 'RBAC parity without ADR-021 split-brain' achievable — none of the drafts identified it. (Note the residual: this path still yields anonymous-<role> provenance unless agentId is also threaded via ctx per the prior improvement — so the two fixes are paired.)
- MAKE THE CONFORMANCE GATE IDENTITY-DRIVEN, NOT SURFACE-DRIVEN. Specify the parity test to drive a role-scoped tool over REST with (i) the correct role -> assert allow + audit row, (ii) a wrong role -> assert DENY, (iii) no/unknown identity -> assert DENY (this is the regression test that pins the fail-open hole closed). A surface-equality test (same tools projected) is necessary but tautological for RBAC; the privilege guarantee requires the wrong-role-denied assertion. Fold this requirement into Draft 1's R0 gate (currently too surface-centric) and keep Drafts 2/3's per-identity framing.
- ADOPT THE PROJECTION CONVENTION AS A DESIGN-HERE ARTIFACT (against Draft 2's deferral). Define the kind->resource / CRUD->method / action->POST {id}/{action} / WATCH->SSE-over-LISTEN-NOTIFY mapping in api/conventions.md at architecture altitude (Drafts 1 and 3 do this; Draft 2 should). Defer only the exact RESOURCE/VERB STRINGS to idea-121. This lets the FIRST banked rung deliver an idiomatic-REST shape (the Director's kube-apiserver ask) rather than a flat `POST /:verb` RPC tunnel that risks reading as MCP re-skinned.
- SPECIFY THE PolicyResult->HTTP UNWRAP CONTRACT in the conventions doc (Draft 2 has it; 1 and 3 must add it). Define: content[0].text |> JSON.parse -> response body; isError -> 4xx/5xx mapping; non-JSON 'prose' handler text -> a defined degraded shape ({message}) flagged for idea-121 typed-results. Without this, the 'thin binding' leaks the MCP envelope and the REST surface is not actually clean REST.
- SYNTHESIZE THE BEST OF THE THREE: Draft 3's risk-sequencing (auth-feasibility spike FIRST, naming-gate forbidding the 'control plane' label until write/actuate ships, read-before-write, NOT-banked spike/register honesty) is the strongest skeleton; Draft 2's mechanism-level grounding (PolicyResult unwrap, the exact deriveContract generalization of computeToolSurfaceRevision folding all-schemas.ts renameMap/watchable, REST_API mounting on the existing Express app reusing requireAuth/bearer-auth) is the most implementation-credible; Draft 1's record-first charter + the dual-binding INVARIANT (every binding terminates in router.handle; no binding hand-lists routes or re-implements RBAC/audit) is the strongest governance framing. Merge: Draft 3's rung sequence + Draft 1's invariant-as-charter + Draft 2's concrete mechanisms + the identity-seam fix above as the explicit keystone.
