## Director's settings question — codifying Workflow/Ultracode approval

- **Can be codified:** Partially. Workflow approval prompts cannot be pre-authorized globally via settings.json as a blanket allowlist. Per-workflow, per-project consent can be recorded and persisted, but Workflow is not a Tool subject to permissions.allow rules like Bash or WebFetch.
- **Recommendation:** For your use-case (trusted operator pre-authorizing autonomous agent orchestration):

**Preferred option if model eligibility allows (and it does not in your current session):**
Set "permissions": {"defaultMode": "auto"} in ~/.claude/settings.json. Auto mode records first-launch consent per workflow in user settings, so future orchestration runs without prompting. Requires Opus 4.6+ or Sonnet 4.6+.

**Current best practice for Haiku 4.5 and other ineligible models:**
Use the Agent SDK or `claude -p` (non-interactive mode) to orchestrate workflows programmatically. This completely bypasses the CLI approval prompt because there is no interactive user to prompt. Host the orchestration as a scheduled routine or background agent that calls the Workflow API directly.

Alternatively, document the one-time "Yes, and don't ask again" selection step for each workflow-per-project in your operator runbook, and have operators execute this setup once before autonomous runs. The consent then persists per project.

**NOT recommended (dangerous, infrastructure risk):**
Bypass permissions mode. It disables all safety checks, not just workflows. Only use in isolated VMs with no network access to production.

**Constraint to note:**
Your current session uses Haiku 4.5, which is NOT eligible for auto mode. Upgrading to Opus 4.6+ or Sonnet 4.6+ would unlock auto mode eligibility if your account/org supports it. Check with your Claude Code admin whether auto mode is enabled in managed settings.
