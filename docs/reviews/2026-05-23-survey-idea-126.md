## §1 Round 1 — Guide intent space

### Q1 — WHY: Envelope adoption scope

| Pick | Option |
|---|---|
| **A ✓** | **Additive-everywhere** — all 20 current SchemaDef kinds gain envelope at once via schema migration |
| B | Opt-in-per-kind — kinds adopt over time; envelope-aware tooling co-evolves |
| C | Pull-forward at first-Mission — kind that needs envelope gets it; backfill at quiet windows |

### Q2 — WHO: Kind discriminator authoring authority

| Pick | Option |
|---|---|
| **A ✓** | **Author-declared at kind registration** — writer declares `spec` / `status` JSON Schema upfront in SchemaDef |
| B | Inferred from existing entity fields — introspection generates `spec` / `status` per kind |
| C | Hybrid — `metadata.*` declared; `spec` / `status` inferred from current fields |

### Q3 — HOW: `apiVersion` evolution scheme

| Pick | Option |
|---|---|
| **A ✓** | **K8s `{group}/{version}` style** (e.g., `core.ois/v1`, `threads.ois/v1`) |
| B | Linear (`v1`, `v2`, `v3`) — no group namespace |
| C | Semver-aware (`v1.0.0` with breaking-vs-additive discipline) |
| D | No versioning at v1 — add only when first breaking change forces it |

### Round 1 Aggregate (Step 3)

> Envelope ships as a substrate-wide all-at-once schema upgrade — all 20 SchemaDef kinds simultaneously, each with architect-authored `spec` / `status` JSON Schemas declared upfront, and `apiVersion` following K8s `{group}/{version}` convention. **Posture: architect-led structural typing with conventional grounding, not opt-in evolutionary growth.**

### Round 1 Per-Question Interpretation (Step 4)

**Tele mapping (this round):** tele-3 + tele-6 primary; all three picks reinforce. No anti-tele drift.

- **Q1-A:** Single-Mission scope across all 20 kind-repository code-paths + one schema-migration script. Mission-83 W5 cutover pattern applies. Architecturally simpler (uniform shape applied at one cutover) but Design-phase prep cost is real (all 20 kinds' schemas authored upfront).
- **Q2-A:** SchemaDef (`hub/scripts/entity-kinds.json` v1.1; 20 kinds LOCKED) extends to carry `spec` / `status` JSON Schemas per kind. No introspection. Existing fields partitioned at Design.
- **Q3-A:** Each kind gets a `group` namespace at migration time. `apiVersion` strictly `<group>/v<N>`. Future bumps: breaking → increment v; additive → stay.

**Cross-question coherence:** Clean composition; reinforced posture; no tensions warranting Round 2 clarification.

## §2 Round 2 — Architect drilled deeper on HOW

### Q1 — Spec/status partitioning convention

| Pick | Option |
|---|---|
| **A ✓** | **Strict K8s convention** — `spec` = desired state (intent); `status` = observed state (runtime / outcome); `metadata` = identity + bookkeeping |
| B | Pragmatic — author-judgment per kind |
| C | Three-tier with explicit rules doc — ratified rules in `docs/methodology/spec-status-partition.md` |

### Q2 — Group taxonomy structure

| Pick | Option |
|---|---|
| **A ✓** | **Minimal 2 groups** — `core.ois` (all current 20 kinds) + `ext.ois` (future extensions) |
| B | Mid-grain 3-5 groups — e.g., `core.ois` / `threads.ois` / `missions.ois` / `methodology.ois` |
| C | Fine-grain per-domain — ~6-8 groups |
| D | Defer to first Mission — architect picks at migration-Design-time |

### Q3 — Migration cutover mechanics

| Pick | Option |
|---|---|
| **A ✓** | **Big-bang substrate-cutover** (mission-83 W5 pattern) — image pre-build + <30s downtime + single PR + schema migration on Hub redeploy |
| B | Dual-write transition — Hub writes both old + new shape for a window |
| C | Forward-compatible additive — add envelope alongside flat fields; no removal at migration |

### Round 2 Aggregate (Step 3)

> Strict K8s `spec` / `status` / `metadata` partitioning + minimal 2-group taxonomy (`core.ois` + `ext.ois`) + big-bang cutover via mission-83 W5 pattern. **No author-discretion on partition; no taxonomy proliferation; no transition window.**

### Round 2 Per-Question Interpretation (Step 4)

**Tele mapping (this round):** A/A/A all reinforce tele-3 + tele-6. No drift.

- **Q1-A:** Author has no discretion at partition level. Per-kind: identity + bookkeeping → `metadata`; declared intent → `spec`; observed/runtime → `status`. Removes the spec/status-partition methodology-doc surface (Q1-C alternative); rules ARE the K8s convention.
- **Q2-A:** All 20 SchemaDef kinds collapse to `core.ois/v1`. `ext.ois` reserved for future extensions (e.g., Class-Tier-Promotion may add Initiative/Concept/Defect under `core.ois` if they belong there, OR a third group emerges if natural fit). Notably simpler than mid-grain or fine-grain — avoids taxonomy-design Design-phase rabbit hole.
- **Q3-A:** Mission-83 W5 pattern proven (image pre-build + <30s effective downtime). Postgres in-place data migration script (read existing flat shape; write envelope shape). Single PR + Hub redeploy.

## §4 Design-phase Inputs (load-bearing for Phase 4)

### 4.1 Architect-side prep (fronts the Mission Design phase)

- **SchemaDef extension** — `hub/scripts/entity-kinds.json` v1.1 → v2.0; per kind, declare `metadata` / `spec` / `status` JSON Schemas
- **Per-kind field partitioning** — for each of the 20 kinds, audit existing fields and assign each to `metadata` / `spec` / `status` per strict K8s convention. Architect-authored at the SchemaDef.

### 4.2 Engineer-side Design surface

- **Migration script** (postgres in-place data migration: read flat shape; write envelope shape; per-kind handlers)
- **Code-path migration** — all kind-repository code-paths (`hub/src/entities/*-repository-substrate.ts`) updated to read/write the new envelope shape
- **`get_resource_shape`-compatibility** — Design checkpoint with idea-121 (M-API-v2.0); the tool's response shape returns `{group: "core.ois", version: "v1", spec-schema, status-schema}` per kind
- **`metadata.sourceThreadId` / `sourceActionId`** — first-class envelope fields; bug-118 fix lands as part of this Mission's migration scope (no separate Mission)

### 4.3 Cutover plan

- **Image pre-build** at quiet pre-cutover window (mission-83 W5 pattern)
- **Single PR + Hub redeploy** at the cutover moment
- **Postgres data migration** in place; schema script in the PR
- **<30s effective downtime budget**
- **Rollback strategy** — image-tag-pin (revert to pre-cutover image); postgres schema migration is forward-only (Design-phase decision on whether reverse migration is needed)

### 4.4 Acceptance criteria

- All 20 kinds carry envelope shape post-cutover (verified via psql JSON shape inspection)
- All Hub MCP tool surfaces return envelope-shaped entities
- `get_resource_shape` returns spec+status JSON Schema per kind (composes with idea-121)
- bug-118 fix verified — new entities filed via any path carry `metadata.sourceThreadId` when calling-session has one
- Downtime ≤30s at cutover
- Old flat-shape data fully migrated (no residual flat entities in postgres)

## §6 Anti-goals (per Survey ratification)

- Does NOT introduce K8s controller-runtime / etcd-watch machinery (K8s *conventions* are the value; *runtime* is not in scope)
- Does NOT pull forward; ships in single big-bang cutover not progressive
- Does NOT auto-migrate per-kind (Q1-A: all 20 at once)
- Does NOT mandate `name` field for entities (optional human-friendly handle per idea-126 body)
- Does NOT subsume the cognitive-tool-surface (idea-121 owns that; envelope is wire-substrate)
- Does NOT add field-depth to cognitive-surface fields without ergonomic check (per thread-628 anti-goal)
- Does NOT introduce dual-write transition window (Q3-A: big-bang only)
- Does NOT proliferate the group taxonomy (Q2-A: minimal `core.ois` + `ext.ois` only)
- Does NOT carry author-discretion on partition (Q1-A R2: strict K8s convention)
- Does NOT split bug-118 fix into a separate Mission (envelope-fix and bug-118-fix share the same Mission scope per §4.2 composition)
